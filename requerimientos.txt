Eres OpenAI Codex y vas a construir un sistema completo para el Área de Compras (multi-rol) como una app Flutter (Dart) usando Firebase (Auth + Firestore + Cloud Functions + FCM + Storage). Objetivo: app muy intuitiva, segura, con flujo de aprobación, trazabilidad por estados, generación de PDF con “checks” de aprobación y notificaciones push con sonido/vibración según rol y estado.

IMPORTANTE
- No inventes pantallas extra innecesarias.
- Prioriza UX limpia, Material 3, navegación clara.
- Implementa arquitectura mantenible (Clean-ish / feature-first) con separación: UI, state management, domain, data.
- Incluye pruebas básicas (unit/widget) donde sea crítico.
- Todo debe quedar ejecutable localmente con instrucciones.

STACK Y DECISIONES (si no se indica lo contrario)
- Flutter estable (última), Material 3.
- State management: Riverpod (preferible) o Bloc si lo justificas (elige uno y sé consistente).
- Firebase:
  - Auth: Email/Password (y deja preparado para SSO futuro).
  - Firestore: principal DB.
  - Cloud Functions: lógica sensible (folio autoincremental, cambios de estado, validaciones, auditoría).
  - FCM: notificaciones push por rol/estado.
  - Storage: PDFs generados y evidencia adjunta si aplica.
- Control de acceso: Reglas Firestore estrictas + verificación extra en Functions.
- Manejo de estados: máquina de estados (enum + transiciones válidas).
- PDF: generar plantilla similar a “evidencias” (a integrar después). Por ahora usa un template fiel con tabla y 3 firmas reemplazadas por “✅ Aprobado” (o check) para Usuario/Compras/Gerencia.

ROLES
- Administradores
- Director General (Gerencia General)
- Compras
- Usuarios (solicitantes)
- Contabilidad

CONCEPTO CLAVE
Cada usuario pertenece a un área/departamento (aprox 12; por confirmar). El usuario NO elige área en UI; viene como propiedad del perfil (user.areaId).

CASO DE USO PRINCIPAL (USUARIO)
1) Login seguro
- Implementa autenticación.
- Reglas básicas de seguridad: validación de sesión, manejo de errores, bloqueo de acciones por rol.
- Prepara verificación de email y política de contraseñas (mínimo razonable).
- Considera logging de eventos relevantes (login, logout, intentos fallidos) sin guardar datos sensibles.

2) Home (Usuario)
Pantalla inicial tras login:
- Botón centrado grande: “Crear orden de compra”
- Debajo: “Ver mi historial de órdenes de compra”
- Tercer botón: NO usar “Proceso”; propon 3 alternativas y elige una por defecto:
  a) “Seguimiento”
  b) “Estatus”
  c) “Ruta de aprobación”
  Implementa con la opción elegida y deja el texto configurable.
- Esquina superior derecha: ícono de perfil:
  - Solo lectura de datos del usuario (nombre, correo, área, rol)
  - Botón cerrar sesión

3) Crear Orden de Compra (Usuario)
Al pulsar “Crear orden de compra”:
- Selector de urgencia: Urgente / Alta / Media / Baja (tipo segmented control o chips).
- Formulario con:
  - Título visible: “Requisición de compra”
  - Folio: autogenerado, formato 000001 en adelante.
    - El contador vive en Firestore y SOLO Admin puede modificarlo (desde consola Firebase, no desde app).
    - El folio se asigna atómicamente (Cloud Function transaction) para evitar duplicados.
  - Nombre solicitante: autocompletado desde usuario
  - Área solicitante: autocompletado desde usuario (en el documento de evidencia lo llaman “proceso”, pero en modelo usar “area”).
  - Fecha: autogenerada (serverTimestamp)
  - Tabla de items (líneas):
    - ITEM (número de fila autoincremental local)
    - Piezas requeridas (int)
    - No. de parte (string)
    - Descripción del producto (string)
    - Cantidad (num)
    - Unidad de medida (string / catálogo)
    - Cliente (string; editable por planeación pero SUJETO A CAMBIOS → modelarlo opcional)
    - Fecha de entrega estimada (date)
- Validaciones: campos requeridos, cantidad > 0, fecha coherente, etc.
- Guardado como “draft” y envío a revisión:
  - El usuario puede guardar borrador.
  - Botón “Enviar a revisión” crea el flujo y notifica a Compras.

4) Historial (Usuario)
- Lista de órdenes del usuario (filtros por estado, fecha, urgencia).
- Cada orden abre detalle con:
  - Información completa
  - Timeline/diagrama de estados (ver punto 5)
  - Botón descargar/ver PDF (si existe)

5) Seguimiento / Estatus / Ruta de aprobación (Usuario)
Pantalla que muestre un diagrama/timeline que avanza según el status de la orden.
Estados propuestos (editables por configuración, pero implementar estos):
- “Por confirmar (Compras)” (pendiente de revisión de compras)
- “Autorizado por Gerencia General”
- “Pago realizado”
- “Orden realizada”
- “Fecha estimada de entrega” (en este punto ya lo visualiza todo mundo y especialmente Contabilidad)
Requisitos del diagrama:
- En cada cambio de estado registrar fecha y hora (serverTimestamp) y quién hizo el cambio (userId, rol).
- Mostrar esos stamps en la UI.
- Esa misma trazabilidad se refleja en el PDF final.
- En cada proceso, Compras o Gerencia pueden “Regresar” la orden al usuario para modificación:
  - Debe registrar motivo (comentario obligatorio).
  - La orden pasa a estado “Requiere cambios (Usuario)”.
  - El usuario edita y reenvía → vuelve a revisión (Compras).
- Notificaciones push con sonido/vibración:
  - A usuario cuando: enviada, requiere cambios, aprobada, etc.
  - A compras cuando: usuario envía o reenvía
  - A gerencia cuando: compras aprueba y requiere autorización
  - A contabilidad cuando: llega a estados relevantes (por ejemplo “Pago realizado”, “Entrega estimada”)
  - Define una matriz de notificación rol x evento.

ACCESO POR ROLES (alto nivel)
- Usuario: crea, edita SOLO cuando está en borrador o “requiere cambios”, ve solo sus órdenes.
- Compras: ve todas o por área (según permisos), confirma/revisa, puede regresar a usuario, puede avanzar a “listo para autorización” de gerencia.
- Gerencia/Director: autoriza o regresa con comentario.
- Contabilidad: ve órdenes a partir de cierto estado (por ejemplo desde “Pago realizado” o “Entrega estimada”, según regla configurable).
- Admin: gestiona usuarios, roles, áreas, contador folio, catálogos (unidades de medida, etc).

PDF FINAL
- El PDF debe ser similar al formato de evidencias (se integrará después), pero implementa un layout profesional:
  - Encabezado con “Requisición de compra”, folio, solicitante, área, fecha, urgencia
  - Tabla de items
  - Sección de firmas: 3 espacios reemplazados por “✅ Aprobado” y datos (nombre/rol/fecha) de:
    - Usuario solicitante
    - Compras
    - Gerencia
- El PDF se genera cuando la orden alcanza un estado “final” (defínelo) y se guarda en Storage + URL en Firestore.
- Permite visualizar/descargar desde la app.

MODELO DE DATOS (Firestore)
Diseña colecciones sugeridas:
- users/{userId}
  - name, email, role, areaId, isActive, createdAt
- areas/{areaId}
  - name, code, isActive
- purchaseOrders/{poId}
  - folio (string “000001”)
  - requesterId, requesterName
  - areaId, areaName
  - urgency (enum)
  - status (enum)
  - createdAt, updatedAt (server)
  - items: array (o subcollection purchaseOrders/{poId}/items)
  - visibility: flags para contabilidad, etc
  - lastReturnReason (optional)
  - pdfUrl (optional)
- purchaseOrders/{poId}/events/{eventId}
  - fromStatus, toStatus, byUserId, byRole, timestamp, comment(optional), type(advance/return/edit)
- counters/folios
  - purchaseOrderNext (int)  (solo modificado por Admin)
- notificationsLog/... (opcional para auditoría)

REGLAS Y SEGURIDAD
- Firestore Security Rules estrictas:
  - Usuario sólo lee sus órdenes y escribe solo en borrador/requiere cambios.
  - Cambios de estado sensibles: SOLO vía Cloud Functions (no desde cliente).
- Cloud Functions:
  - assignFolioAndCreateOrder (transaction)
  - transitionStatus (valida rol + transición + registra evento + dispara notificaciones)
  - returnToUser (requiere comentario)
  - generatePdfOnFinalState (trigger on status)
- Auditoría: registrar timestamps server y actor.
